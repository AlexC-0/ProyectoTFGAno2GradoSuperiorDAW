1. OBJETIVO DEL DOCUMENTO


====================================================================


Este documento define el mapa de navegacion actual del TFG GR-Inn / ECO & WOODS. Su finalidad es mostrar de forma ordenada como se conectan las pantallas principales de la aplicacion, que rutas puede seguir cada tipo de usuario y que acciones forman parte del funcionamiento interno del sistema.

El mapa diferencia tres perfiles principales:


- Visitante no autenticado.

- Usuario registrado.

- Usuario administrador.


Tambien separa las pantallas visibles de las rutas de accion, ya que algunas paginas PHP no representan una pantalla completa, sino que procesan operaciones como anadir productos al carrito, modificar favoritos, eliminar elementos o cerrar sesion.

====================================================================


El mapa representa el estado actual del proyecto, no una version futura ni idealizada.

3. LEYENDA UTILIZADA


====================================================================


->              Navegacion directa entre pantallas.

[accion]        Accion del usuario dentro de una pantalla.

[POST]          Envio de formulario que modifica datos.
[AJAX]          Accion asincrona sin recarga completa de pagina.

[requiere sesion] El usuario debe estar autenticado.
[solo admin]    Acceso exclusivo para usuario administrador.

[secundaria]    Pantalla existente, pero no central en el flujo principal.

[sin pantalla]  Ruta tecnica de procesamiento, no vista completa.

[condicion]     Regla que cambia el flujo segun usuario, sesion o permisos.

4. ROLES CONTEMPLADOS


====================================================================


4.1. Visitante no autenticado


Puede consultar la pantalla de inicio, acceder al login, registrarse y explorar contenido publico si llega a los catalogos o detalles. No puede publicar, contactar, guardar favoritos, escribir resenas ni mantener un carrito asociado a cuenta.

4.2. Usuario registrado


Puede navegar por la aplicacion completa orientada a usuario final. Tiene acceso a catalogos, detalles, favoritos, carrito, publicacion de muebles, resenas, mensajeria y gestion de perfil.

4.3. Usuario administrador


Hereda las funciones del usuario registrado y anade gestion global de contenido. Puede entrar al panel de administracion y publicar tanto muebles como recambios 3D.

13. RESTRICCIONES Y CONDICIONES DE NAVEGACION


====================================================================


13.1. Condiciones por sesion


- Las acciones privadas requieren usuario autenticado.

- El visitante puede explorar, pero no gestionar datos personales ni realizar operaciones de usuario.

- El carrito real esta asociado a sesion de usuario.

- El perfil solo existe para usuarios autenticados.


13.2. Condiciones por rol


- El usuario normal puede publicar muebles, pero no recambios 3D.

- El administrador puede publicar muebles y recambios 3D.

- El administrador tiene acceso a Panel Admin.

- El usuario normal no puede acceder a admin.php.


13.3. Condiciones sobre acciones


- Las acciones AJAX o POST modifican datos sin representar pantallas completas.

- Al actualizar favoritos o carrito, la interfaz debe reflejar el cambio con aviso o actualizacion de contador.

- Las eliminaciones actualizan la pantalla de origen.

- La compra termina en modo prueba.

15. CRITERIO DE FIDELIDAD PARA LA MEMORIA DEL TFG


====================================================================


Este mapa debe entenderse como una representacion funcional del TFG actual. Para mantener la fidelidad con el proyecto, la documentacion debe respetar estos puntos:

- La cabecera cambia segun sesion y rol.

- El visitante tiene menu reducido.

- El usuario registrado accede a catalogos, carrito, favoritos, mensajes, resenas, publicacion de muebles y perfil.

- El administrador hereda funciones de usuario y anade Panel Admin y publicacion de recambios.

- Mi perfil concentra la gestion personal.

- Panel Admin concentra la gestion global.

- El carrito existe y finaliza compras en modo prueba.

- El sistema de pago real no forma parte del flujo actual.

- Las rutas AJAX y POST son acciones funcionales, no pantallas independientes.

- Las pantallas secundarias existen, pero no son el centro del recorrido principal.
